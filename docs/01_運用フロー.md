# 01. 運用フロー

## 目的・背景

属人化しがちだった「電子申告データチェック・製本・郵送」の一連業務を、総務パート等へ安全かつスムーズに移管する。
**業務ミス（事故）を防ぎつつ、担当者の負担を軽減する**ことを目的とする。

## 3ツールの役割分担

| ツール | 役割 |
|---|---|
| **Googleフォーム** | 依頼の入口。必須項目＋選択式で記入漏れ・表記ゆれを構造的に防ぐ |
| **スプレッドシート** | フォーム回答が自動で溜まる管理台帳。ステータス・期限・完了報告で進捗を可視化 |
| **チャットワーク** | フォーム送信の通知 → パートがリアクションで受託 → 郵送前に担当者へ最終確認 |

## フロー

```
1. 担当者：達人へのデータ取込を完了する
        ↓
2. 担当者：Googleフォームで依頼（必須項目を入力）
        ↓ （回答が台帳に自動転記）
3. チャットワークに依頼が通知される
        ↓
4. パート：リアクションで受託を明確化
        ↓
5. パート：受託時チェック → 電子申告 / 製本 を実施（台帳ステータス更新）
        ↓
6. 郵送前：パートが担当者へ最終確認（誤発送防止）＝ ステータス「確認待ち」
        ↓
7. 担当者：OK → パートが発送 → 完了報告（郵送日・追跡番号・控え保管）を台帳に記録
```

## パート受託時チェック項目（作業前に必ず確認）

- [ ] 顧問先名・コードの確認（依頼内容とデータの一致）
- [ ] 対象年度・期の確認（過去データとの誤認防止）← フォームの「対象事業年度・期」で担保
- [ ] 指定税目のデータ有無（法人税・消費税などが揃っているか）
- [ ] 決算書データの有無（財務諸表が正しく連動しているか）
- [ ] 地方税の申告先（自治体）の確認（23区外・複数支店の漏れ防止）← フォームの「申告先自治体」で担保

## ステータス定義

タスク単位で管理する（1依頼でも「申告のみ」「申告＋製本＋郵送」と中身が変わるため）。

| タスク | ステータス値 |
|---|---|
| **申告** | 不要 ／ 依頼中 ／ 完了 |
| **製本** | 不要 ／ 依頼中 ／ 完了 |
| **郵送** | 不要 ／ 依頼中 ／ **確認待ち** ／ 完了 |

- **「不要（対象外）」を必ず用意**：製本・郵送が不要な依頼を「依頼中」のまま放置して見えるのを防ぐ。
- **郵送の「確認待ち」**：パート案の肝である「郵送前に必ず担当者へ最終確認」を台帳上で担保するための1段階。担当者OK後に「完了」へ。

### 全体完了の定義

申告・製本・郵送の3タスクが、それぞれ「完了」または「不要」になった時点で、その依頼を全体完了とする。

## 役割分担（要決定）

- チャットのタスクを「**依頼者自身に立てる**」か「**パートに立てる**」か → 運用ルールとして確定が必要。
- 依頼できるのは誰か（担当者のみ／全社員）、パートは何人で受託をどう割り振るか。
- 郵送前の最終確認は、依頼者本人が行う（戻し先＝フォームの「依頼者」）。
